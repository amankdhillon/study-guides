The Big Picture: Why GPU Programming is Different

🏭 Think of it Like a Factory

CPU Programming: You have one super skilled worker who does tasks one at a time, very fast.

GPU Programming: You have thousands of workers who all do simple tasks simultaneously. None are as skilled as the CPU worker, but together they're incredibly powerful.

In OpenCL, we organize these "workers" in a specific way:

Work-Item (Thread)

The individual worker. Each one executes the same program but on different data. Think: "I'm worker #53, so I process element #53 of the array."

Work-Group (Block)

A team of workers. They can share a workspace (local memory) and coordinate with each other. Typically 64-256 workers per team.

NDRange (Grid)

The entire workforce. All teams working on the complete job. Can be 1D (a line), 2D (a grid), or 3D (a cube) depending on your problem.

Understanding Work-Item IDs

Every work-item needs to know: "Which piece of data should I process?" This is determined by its ID.

The Golden Formula

The most important equation in this entire course:

global_id = group_id × local_size + local_id

Why This Formula Works

Think about it in parts:

  • group_id × local_size = How many work-items are in ALL previous groups?
  • + local_id = Add your position within YOUR group
  • = global_id = Your unique position across ALL work-items!

OpenCL Functions Cheat Sheet

Function Returns Example Think of it as...
get_global_id(0) Your unique ID Returns: 0 to N-1 "What's my employee number?"
get_local_id(0) Your position in your team Returns: 0 to (team_size-1) "What's my position on my team?"
get_group_id(0) Which team you're on Returns: 0 to (num_teams-1) "Which team am I on?"
get_local_size(0) How big is your team Returns: team size (e.g. 128) "How many people on my team?"
get_global_size(0) Total workers Returns: total work-items "How many total employees?"
get_num_groups(0) Total teams Returns: number of work-groups "How many teams total?"

Why 2D? The Image/Matrix Problem

🖼️ Real-World Example

Imagine you need to process a 1920×1080 image. That's over 2 million pixels! Each pixel needs work done on it. How do you organize your workers?

Option 1D: Line up 2 million workers in a row. Worker #1,234,567 processes pixel #1,234,567.

Option 2D: Organize workers in a 1920×1080 grid. Worker at position (row=500, col=800) processes the pixel at row 500, column 800.

Option 2D is WAY more natural for grid-like data!

The Formula

To access element at (row, col) in a matrix with W columns:

index = row × W + col

Why?

  • row × W = Skip all previous rows (each row has W elements)
  • + col = Move to the right column within that row

Grid Sizing: The Ceiling Division Trick

One of the trickiest parts: How many work-groups do you need?

⚠️ Common Mistake: Using regular division!
// ❌ WRONG - might not process all elements! num_groups = N / block_size; // Floor division // ✅ CORRECT - ceiling division num_groups = (N + block_size - 1) / block_size;

Why This Works

Let's say N = 1000, block_size = 256:

  • Wrong way: 1000 / 256 = 3 (floor) = 768 work-items. We miss 232 elements!
  • Right way: (1000 + 255) / 256 = 1255 / 256 = 4 (floor) = 1024 work-items. ✓

The extra threads (1024 - 1000 = 24) just need boundary checks to not process invalid data.

The Memory Speed Pyramid

🏢 Office Building Analogy

Private Memory (Registers): Items on your desk. Instant access, but tiny space, only YOU can use it.

Local Memory (Shared): Shared filing cabinet for your team. Fast access, limited space, your whole team can use it.

Global Memory (DRAM): Warehouse across town. Huge storage, but slow to access, everyone in the company uses it.

Constant Memory: Printed handbook. Read-only, cached for speed, same for everyone.

🚀 Private Memory (Fastest)

• Each work-item's registers

• Tiny (~few KB per work-item)

• Access time: ~1 cycle

• Usage: Local variables automatically go here

⚡ Local Memory (Very Fast)

• Shared within work-group

• Small (~16-64 KB per work-group)

• Access time: ~10 cycles

• Usage: Declare with __local keyword

🐢 Global Memory (Slow)

• Accessible by all work-items

• Huge (several GB)

• Access time: ~400-800 cycles

• Usage: Main data storage (__global)

📖 Constant Memory (Cached)

• Read-only

• Small (~64 KB)

• Access time: ~1-10 cycles (if cached)

• Usage: Parameters (__constant)

🔑 The Big Idea: The whole point of tiling and optimization is to load data from slow global memory into fast local memory, then reuse it many times!

Barriers: The Coordination Checkpoint

Barriers are synchronization points. ALL work-items in a work-group must reach the barrier before ANY can proceed.

Threads arriving at barrier...

Phase 1: Loading Data

Each thread loads its data into shared memory

shared[get_local_id(0)] = input[get_global_id(0)];
Phase 2: BARRIER

Wait for ALL threads to finish loading!

barrier(CLK_LOCAL_MEM_FENCE); // ⚠️ Everyone waits here!
Phase 3: Using Data

Now it's safe to read what other threads wrote

float neighbor = shared[get_local_id(0) + 1]; // Safe!
⚠️ THE DEADLOCK TRAP
// ❌ DEADLOCK! Never put barrier in conditional! if (get_local_id(0) < 10) { barrier(CLK_LOCAL_MEM_FENCE); // Only threads 0-9 hit this! } // Threads 10+ never reach barrier, threads 0-9 wait forever ☠️ // ✅ CORRECT - everyone hits the barrier if (get_local_id(0) < 10) { // do special work } barrier(CLK_LOCAL_MEM_FENCE); // All threads hit this ✓

When Do You Need a Barrier?

Use this simple checklist:

The 3 Questions

  1. Does Thread A write to shared (__local) memory?
  2. Does Thread B read from that same location?
  3. Could Thread B read before Thread A finishes writing?

If YES to all three → YOU NEED A BARRIER!

🧮 Practice: Find the Bug

This code has a race condition. Can you spot where the barrier should go?

__local float data[256]; int tid = get_local_id(0); // Step 1: Everyone loads their data data[tid] = input[get_global_id(0)]; // Step 2: Sum with neighbor - BUG HERE! 🐛 if (tid > 0) { float sum = data[tid] + data[tid - 1]; }

💡 Solution

The Problem: Thread 1 tries to read data[0], but Thread 0 might not have written it yet!

The Fix:

data[tid] = input[get_global_id(0)]; barrier(CLK_LOCAL_MEM_FENCE); // ⚠️ Add barrier here! if (tid > 0) { float sum = data[tid] + data[tid - 1]; // Now safe! }

✅ Quick Check: Barriers

Question: Which code snippet REQUIRES a barrier?

A) 
float temp = data[tid];
result[tid] = temp * 2;
B) 
shared[tid] = data[tid];
float sum = shared[tid] + shared[tid+1];
C) 
int x = get_global_id(0);
output[x] = input[x] + 1;

💡 Explanation

Answer B needs a barrier!

Thread tid writes to shared[tid], then thread tid tries to read shared[tid+1] which was written by thread tid+1. Without a barrier, the read might happen before the write!

The correct code:

shared[tid] = data[tid];
barrier(CLK_LOCAL_MEM_FENCE);  // ← Need this!
float sum = shared[tid] + shared[tid+1];
